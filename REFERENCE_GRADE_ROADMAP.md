# Reference-Grade Roadmap

## Goal

Move the repository from an elite draft foundation to a final published reference
encyclopedia that is accurate, maintainable, useful, and safe to publish.

## What “Reference-Grade” Means

Reference-grade does not mean long. It means dependable.

A reference-grade page has:

- Clear scope
- Correct terminology
- Practical examples
- Primary or reputable references
- Defensive value
- Safe publication boundaries
- Maintenance owner
- Review date

## Phase 1 — Editorial System

Deliverables:

- Publishing guide
- Editorial style guide
- Source and citation policy
- Reference-grade standard
- Human review checklist
- Content gap audit

Definition of done:

- Every page can be judged consistently.
- Draft pages are not confused with published pages.
- Reviewers know what to check.

## Phase 2 — Canonical Reference Pages

Create a small set of high-quality model pages:

- Password spray detection reference
- Account compromise response reference
- Log4Shell reference
- CloudTrail IAM detection reference
- Broken Object Level Authorization reference
- Prompt injection defense reference
- Security operations knowledge platform reference

Definition of done:

- These pages set the writing and technical standard for the rest of the repo.

## Phase 3 — Artifact Libraries

Build concrete libraries:

- 20 detection rules with metadata and sample events
- 10 CVE writeups
- 10 IOC reports
- 10 incident response playbooks
- 10 architecture diagrams
- 5 automation utilities

Definition of done:

- The repository contains reusable artifacts, not only explanations.

## Phase 4 — Review and Publication

Tasks:

- Verify sources.
- Promote selected pages to `reviewed`.
- Add diagrams.
- Add link checks.
- Add JSON schema validation.
- Add issue-based maintenance workflow.

Definition of done:

- The repository has a curated published index.
- Every published page has a review date.

## Phase 5 — Maintenance

Ongoing rhythm:

- Weekly: update one page or close one gap.
- Monthly: review one domain.
- Quarterly: archive stale content and refresh references.
- After incidents: update detections, playbooks, and architecture notes.
